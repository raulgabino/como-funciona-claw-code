Como funciona
Claude Code

Un curso interactivo para entender la ingenieria de un agente de IA de codigo — la maquinaria invisible que conecta herramientas, orquesta tareas y gestiona el contexto entre tu prompt y la respuesta.

Antes de ver el codigo,
entremos a un restaurante.

Imagina un restaurante con 200 platillos en la carta. Tu llegas, te sientas en la barra, y haces tu pedido. No ves la cocina. Solo ves que, despues de un rato, tu plato aparece.

Pero detras de esa barra hay un sistema completo: un chef ejecutivo que coordina todo, 45 utensilios especializados, un inspector de sanidad con 6 niveles de revision, una bodega con memoria de cada comensal, y un chef nocturno que trabaja cuando cierran.

Claude Code funciona exactamente asi. Y en este curso, vamos a recorrer cada rincon de esa cocina.

En cada modulo de este curso, volveremos a este restaurante para anclar cada concepto tecnico en algo familiar.

Modulo 1

Que es Claude Code
por dentro?

Un sistema de agente de codigo con ~512,000 lineas de TypeScript que conecta a Claude con tu entorno.

En el restaurante: Este restaurante tiene 200 platillos en la carta, 45 utensilios especializados de cocina, y 6 chefs con roles diferentes. Tu solo ves la barra donde pides — pero detras hay una operacion completa.

Cuando usas Claude Code, escribes un prompt y recibes una respuesta. Pero entre tu texto y la respuesta hay un sistema masivo: ~512,000 lineas de TypeScript distribuidas en ~1,884 archivos que deciden que herramientas usar, que permisos revisar, y como gestionar la conversacion.

🔧
45 herramientas en 9 categorias
Files, Shell, Web, Agents, Planning, Tasks, Scheduling, UI, MCP — cada una especializada en un tipo de operacion.
🔒
6 modos de permisos
Desde "preguntar todo" hasta "modo auto" con un clasificador de IA de dos etapas que decide que es seguro ejecutar.
⚡
200K tokens de contexto
Con auto-compactacion cuando quedan ~13K tokens, micro-compactacion de resultados viejos, y hasta 1M para modelos avanzados.
Por que importa?

Entender como funciona Claude Code por dentro te permite dirigirlo mejor. Saber que tiene herramientas dedicadas (Read en vez de cat, Grep en vez de grep) o que su system prompt tiene 6 capas de prioridad cambia como le hablas.

Modulo 2

El system prompt —
las reglas de la casa

6 capas de prioridad, 7 secciones estaticas, y un cache boundary que separa lo fijo de lo dinamico.

En el restaurante: Las reglas escritas en la pared de la cocina. Las del dueno van arriba (nadie las cambia), las del turno van abajo (se actualizan cada dia). Y hay unas reglas especiales — las del CLAUDE.md — que el chef consulta como su recetario personal.

El system prompt no es un solo bloque de texto. Es un sistema de 6 capas de prioridad donde cada capa puede reemplazar o complementar a las demas. La capa mas alta gana.

Capas 1-3 — Overrides (mayor prioridad)
1. Override prompt 2. Coordinator (multi-agente) 3. Agent (sub-agente) 4. Custom (--system-prompt)
Capa 5 — Default (7 secciones + CLAUDE.md)
Intro + Seguridad System Doing Tasks Executing Actions Using Tools Tone & Style Output Efficiency
Capa 6 — Append (contexto dinamico)
Memoria (MEMORY.md) MCP servers Idioma detectado Entorno (OS, shell, git)
Estructura de capas
// Prioridad del system prompt
const layers = [
  "override",     // reemplaza TODO
  "coordinator",  // modo multi-agente
  "agent",        // sub-agente
  "custom",       // flag CLI
  "default",      // 7 secciones + CLAUDE.md
  "append",       // memoria + entorno
];
// Cache boundary: entre seccion 4 y 5
// Secciones 1-4 se cachean (estaticas)
// Secciones 5-7 + append son dinamicas
En espanol
📍 Las instrucciones de Claude Code vienen en capas:
La capa mas alta reemplaza todo lo de abajo.
En modo normal, solo se usa "default" + "append".
 
La capa default tiene 7 secciones fijas
+ el contenido de CLAUDE.md del proyecto.
 
Hay un "cache boundary" (corte de cache):
las primeras 4 secciones se reusan entre llamadas
para ahorrar costos de API.
Por que CLAUDE.md tiene tanto poder?

CLAUDE.md se inyecta en la capa 5 (default), no en la capa 6 (append) como la memoria regular. Esto le da mas peso porque esta mas arriba en la jerarquia. Es como las reglas del chef principal vs las notas del mesero — ambas cuentan, pero las del chef mandan.

Regla de herramientas dedicadas
// El system prompt dice explicitamente:
// "Do NOT use Bash to run commands
//  when a relevant dedicated tool
//  is provided."

// En vez de:
Bash("cat src/index.ts")     // MAL
Bash("grep -r 'TODO' .")    // MAL
Bash("find . -name '*.ts'") // MAL

// Claude Code usa:
Read("src/index.ts")         // BIEN
Grep("TODO")                 // BIEN
Glob("**/*.ts")              // BIEN
En espanol
📍 Regla fundamental del system prompt:
"No uses Bash cuando hay una herramienta dedicada."
 
En vez de usar cat para leer archivos,
usar grep del shell para buscar texto,
o find para buscar archivos...
 
Claude Code usa Read, Grep y Glob.
Son mas seguros, mas rapidos de revisar,
y el usuario puede aprobar/denegar cada uno.
Herramientas condicionales

No todas las 45 herramientas estan siempre disponibles. SleepTool solo aparece en modo KAIROS. CronTools solo con AGENT_TRIGGERS habilitado. WebBrowserTool y MonitorTool son internos de Anthropic. Las herramientas se inyectan condicionalmente segun el contexto.

Modulo 4

Permisos y seguridad

6 modos de permisos y un YOLO classifier de dos etapas que decide que es seguro.

En el restaurante: El inspector de sanidad con 6 niveles de revision. Puede ser laxo ("solo reviso si huele raro"), estricto ("reviso cada ingrediente"), o puede tener un asistente con IA que clasifica riesgos automaticamente en dos pasadas.
1
default — Pregunta al usuario en cada operacion sensible. El modo mas conservador y seguro.
2
acceptEdits — Auto-aprueba lectura y escritura de archivos. Shell y red siguen pidiendo permiso.
3
bypassPermissions — Auto-aprueba TODO sin preguntar. Sin checks de seguridad. Alto riesgo.
4
dontAsk — En vez de preguntar, deniega automaticamente. Nunca interrumpe pero puede fallar en silencio.
5
auto — El YOLO classifier decide allow/block sin preguntar. El modo con clasificador de IA.
6
plan — Modo de planificacion. Acceso restringido: solo lectura y busqueda, sin edicion ni ejecucion.
// YOLO Classifier — 2 etapas de decision 
// Stage 1: Decision rapida (max 64 tokens)
// Pregunta: "Es seguro ejecutar esto?"
// Si la respuesta es "si" -> ejecutar
// Si la respuesta es "no" -> ir a Stage 2

// Stage 2: Razonamiento profundo (max 4096 tokens)
// Analiza el comando completo con contexto
// Si aun bloquea -> denegar la operacion

// Patrones SIEMPRE bloqueados (hardcoded):
const ALWAYS_BLOCKED = [
  "python", "node", "deno", "ruby", "perl",
  "npx", "bunx", "bash", "sh", "ssh",
  "eval", "exec", "sudo", "curl", "wget",
  "kubectl", "aws", "gcloud", "git", "gh"
];
// Fallback: si demasiados bloqueos consecutivos,
// vuelve a modo interactivo automaticamente
Defensa en profundidad

El sistema tiene multiples capas: los 6 modos definen la politica general, el YOLO classifier evalua cada accion individual en modo auto, y los patrones hardcoded bloquean comandos peligrosos sin importar el modo. Si una capa falla, las otras te cubren.

Modulo 5

Contexto y

200K tokens de context window, auto-compactacion, y micro-compactacion de resultados.

En el restaurante: La mesa del chef tiene espacio limitado — solo caben cierta cantidad de platos al mismo tiempo. Cuando se llena, el ayudante archiva los platos terminados, deja un resumen ("mesa 4: entraron 3 platillos, salieron bien"), y libera espacio para los nuevos.

Micro-compactacion

Ademas de la auto-compactacion (que resume toda la conversacion), Claude Code hace micro-compactacion: reemplaza los resultados viejos de herramientas con [Old tool result cleared]. Asi libera espacio de forma incremental sin necesidad de comprimir toda la historia.

Limites de salida

Claude Code maneja 4 niveles de output: 32K tokens por defecto, 8K en modo breve (capped), 64K cuando escala automaticamente (escalated), y 20K para el resumen de compactacion. El circuit breaker detiene la compactacion despues de 3 fallos consecutivos.

Escenario de diagnostico
Llevas 2 horas en una sesion y de pronto Claude parece "olvidar" instrucciones que le diste al inicio. Que esta pasando?
Modulo 6

Memoria y CLAUDE.md

4 tipos de memoria persistente y un archivo de configuracion que pesa mas que el resto.

En el restaurante: El libro de recetas personales del chef — lo consulta al inicio de cada turno. Tiene notas sobre clientes ("al de la mesa 4 no le gusta el cilantro"), feedback del inspector ("usar guantes con el sushi"), y referencias a proveedores.
👤
user
Info sobre el usuario: rol, preferencias, conocimientos. Ayuda a personalizar las respuestas y el nivel de detalle.
💬
feedback
Correcciones y confirmaciones del usuario. "No hagas X" o "Asi si, sigue haciendolo". Evita repetir errores entre sesiones.
📌
project
Info sobre el trabajo en curso: plazos, decisiones, contexto de negocio. Se inyecta como parte del append prompt.
🔗
reference
Punteros a recursos externos: "los bugs estan en Linear proyecto INGEST", "el dashboard es grafana.internal/d/api-latency".

En espanol
📍 La memoria vive en archivos markdown:

Cada memoria tiene su propio archivo

CLAUDE.md vs MEMORY.md

CLAUDE.md se inyecta en la capa 5 (default) del system prompt — junto con las 7 secciones estaticas. MEMORY.md se inyecta en la capa 6 (append). Esto significa que las instrucciones de CLAUDE.md tienen prioridad sobre la memoria. Usa CLAUDE.md para reglas permanentes del proyecto y la memoria para contexto cambiante.

Modulo 7

Modo proactivo
KAIROS

El modo autonomo donde Claude trabaja sin input del usuario — con SleepTool obligatorio.

En el restaurante: El chef que cocina solo de noche cuando el restaurante cierra. Revisa el inventario, prepara la masa para manana, limpia las estaciones. Pero tiene un reloj obligatorio — cada 30 minutos DEBE descansar. Si no, lo sacan de la cocina.

KAIROS reemplaza el prompt default por uno de modo autonomo. En vez de esperar instrucciones del usuario, Claude ejecuta un tick loop continuo: revisa tareas, ejecuta, y duerme.

1
Check — Revisa la cola de tareas pendientes. Si hay algo por hacer, continua. Si no, llama a SleepTool.
2
Execute — Ejecuta la siguiente accion usando las herramientas disponibles. Mismo sistema de permisos aplica.
3
Sleep — Obligatorio. Claude DEBE llamar a SleepTool entre ciclos. Sin esta pausa, el sistema lo detiene.
4
Report — Documenta resultados y progreso. Luego vuelve al paso 1. El loop continua hasta completar o ser detenido.
Tarea creada
→
KAIROS pickup
→
Execute
→
SleepTool
→
Check again
→
Complete
Por que SleepTool es obligatorio?

Sin SleepTool, un agente autonomo podria consumir recursos infinitamente en un loop sin salida. Es un circuit breaker por diseno — fuerza pausas que permiten al sistema verificar que todo va bien, y al usuario intervenir si algo salio mal. Es como la ley que obliga a los pilotos a descansar entre vuelos.

CLAUDE.md + 4 tipos

10 tips para dirigir mejor a Claude Code

1
Usa CLAUDE.md
Pon instrucciones permanentes del proyecto en CLAUDE.md — se inyecta en la capa 5 con mayor prioridad que la memoria.
2
Nombra herramientas
Di "usa Read para leer el archivo" en vez de "lee el archivo". Las herramientas dedicadas son mas seguras que Bash.
3
Entiende los modos
Empieza en "default", sube a "acceptEdits" cuando confies, usa "auto" solo si entiendes que YOLO bloquea.
4
Vigila el contexto
En sesiones largas, repite instrucciones clave. La auto-compactacion puede resumir tus instrucciones originales.
5
Usa sub-agentes
Para tareas paralelas, pide que lance Agent tools. Cada sub-agente tiene su propio contexto y puede trabajar en aislamiento.
6
Modo plan primero
Para cambios grandes, usa /plan antes de implementar. Modo plan restringe herramientas a solo lectura para revisar sin riesgo.
7
MCP para integraciones
Conecta servicios externos (Slack, GitHub, bases de datos) via MCP. Las herramientas se inyectan dinamicamente.
8
Memoria estrategica
Guarda solo lo que no se puede derivar del codigo. Preferencias, decisiones de negocio, feedback — no patrones de codigo.
9
Evita patrones bloqueados
En modo auto, python/node/sudo/curl estan siempre bloqueados. Si necesitas ejecutarlos, usa modo interactivo.
10
Diagnostica con arquitectura
Si algo falla, piensa en capas: es un problema de prompt? De permisos? De contexto agotado? De memoria desactualizada?
Quiz final: Escenario de arquitectura
Un colega puso instrucciones importantes en MEMORY.md pero Claude parece ignorarlas cuando hay conflicto con otras instrucciones. Cual es la explicacion arquitectonica?
Para llevar

Claude Code no es magia — es ingenieria de sistemas con patrones claros: prompts en capas, herramientas especializadas, permisos multinivel, gestion de contexto con compactacion, memoria persistente, y un modo autonomo con circuit breakers. Entender estos patrones te convierte en un operador mas efectivo de cualquier agente de IA.
