Antes de ver el código,
entremos a un restaurante.

Imagina un restaurante con 200 platillos en la carta. Tú llegas, te sientas en la barra, y haces tu pedido. No ves la cocina. Solo ves que, después de un rato, tu plato aparece.

Pero detrás de esa barra hay un sistema completo: un chef ejecutivo que coordina todo, una bodega con inventario de cada ingrediente, un inspector de sanidad que revisa que nada peligroso llegue al plato, y un archivo donde queda registrada cada comanda del día.

Claw Code funciona exactamente así. Y en este curso, vamos a recorrer cada rincón de esa cocina.

En cada módulo de este curso, volveremos a este restaurante para anclar cada concepto técnico en algo familiar.

Módulo 1

¿Qué es Claw Code y
para qué sirve?

Un clean-room rewrite de los patrones arquitectónicos de Claude Code.

En el restaurante: Alguien vio un restaurante famoso desde afuera, entró a la cocina, tomó notas de cómo estaba organizada — y luego construyó su propia cocina desde cero, sin copiar las recetas. Eso es Claw Code.

Imagina que Claude Code es un coche de carreras. Tú ves la interfaz — la terminal, los comandos, las respuestas. Pero debajo del capó hay un motor complejo: el harness.

Claw Code desarma ese motor pieza por pieza, lo reconstruye en Python puro, y ahora lo está portando a Rust para hacerlo más rápido y seguro.

🔬
Ingeniería inversa
Mapea los 207 comandos y 184 herramientas del Claude Code original como "snapshots" JSON.
🔧
No es una copia
Cada módulo es una reimplementación desde cero — los "shims" replican el comportamiento, no el código.
🦀
Rumbo a Rust
El port a Rust añade seguridad de memoria, streaming SSE nativo, y un cliente OAuth integrado.
💡 ¿Por qué importa?

Un harness de agente es lo que convierte un LLM "desnudo" (que solo produce texto) en una herramienta que puede ejecutar comandos, editar archivos, y conectarse a servicios externos. Entender cómo funciona te permite dirigir mejor a cualquier agente de IA.

Los personajes
del sistema

Como una orquesta sinfónica, cada módulo tiene un papel definido.

En el restaurante: Tres zonas claras — la barra donde pides (CLI), la cocina donde se coordina todo (Runtime), y la bodega donde se guardan ingredientes y recetas (Datos).

En español

Módulo 3

El Runtime —
director de orquesta

Cuando escribes un prompt, el runtime orquesta todo lo que pasa debajo.

En el restaurante: El chef ejecutivo. Nunca cocina directamente, pero lee cada comanda, decide qué estación la prepara, y se asegura de que nada salga sin su visto bueno.

El PortRuntime es el cerebro del sistema. Cuando llega un prompt, ejecuta una coreografía precisa en siete pasos.

1
Prefetch — Arranca procesos en paralelo: escaneo del proyecto, lectura de keychain, verificación MDM.
2
Context — Mapea el espacio de trabajo: cuántos archivos Python hay, si existe un archivo TypeScript original para comparar.
3
Routing — Tokeniza tu prompt, lo compara contra 207 comandos y 184 herramientas, y puntúa los mejores candidatos.
4
Permission check — Filtra herramientas peligrosas (como bash destructivo) antes de ejecutar nada.
5
Execution — Ejecuta los comandos y herramientas seleccionados a través del ExecutionRegistry.
6
Query Engine — Procesa el mensaje, gestiona el presupuesto de tokens, y compacta la historia si es necesario.
7
Persist — Guarda la sesión en disco para poder retomarla después.
Código — runtime.py
def route_prompt(self, prompt, limit=5):
    tokens = {t.lower() for t in
        prompt.replace('/',' ')
              .replace('-',' ')
              .split() if t}
    by_kind = {
      'command': self._collect_matches(
          tokens, PORTED_COMMANDS, 'command'),
      'tool': self._collect_matches(
          tokens, PORTED_TOOLS, 'tool'),
    }
En español
📍 Cuando llega un prompt, lo despedaza:
rompe barras y guiones en palabras sueltas,
las pone en minúsculas,
y elimina vacías.
Luego busca coincidencias en dos catálogos:
→ el catálogo de comandos (207 entradas)
→ el catálogo de herramientas (184 entradas)
💡 Concepto clave: Routing por tokens

Esto es como el sistema de correo postal: tu prompt llega, la oficina de clasificación lo descompone en palabras clave, y cada palabra se compara contra las "direcciones" (nombres, descripciones) de todos los comandos y herramientas disponibles. El que tenga más coincidencias gana.

Módulo 4

Los datos de referencia —
el inventario maestro

Todo lo que Claw Code sabe sobre Claude Code vive en archivos JSON.

En el restaurante: La carta impresa con los 200 platillos. No es una receta ejecutable — es un catálogo descriptivo. Cada mañana, el chef la memoriza para saber qué puede ofrecer.

El directorio reference_data/ es la bodega del sistema. Contiene snapshots del Claude Code original — no código copiado, sino un catálogo exhaustivo de lo que cada pieza hacía.

📋
commands_snapshot.json
207 entradas de comandos mapeados desde los archivos TypeScript originales — add-dir, agents, autofix-pr, y más.
🔧
tools_snapshot.json
184 entradas de herramientas — AgentTool, BashTool, FileReadTool, agentMemory, MCP integrations, etc.
📁
subsystems/*.json
Desglose por subsistema: skills, remote, voice, server, memdir, components, outputStyles, native_ts.
// Ejemplo de una entrada en tools_snapshot.json 
{
  "name": "BashTool",
  "responsibility": "Tool module mirrored from archived TypeScript path tools/BashTool",
  "source_hint": "tools/BashTool.ts"
}

Estos archivos no ejecutan nada — son pura metadata. Cuando el sistema arranca, los carga en memoria como tuplas de PortingModule y los usa para routing, auditoría de paridad, y generación de reportes.

🎯 Patrón de diseño: Catálogo estático

Separar los datos de la lógica es como tener un directorio telefónico independiente de la central telefónica. Puedes actualizar el catálogo de comandos sin tocar el código que los ejecuta. Esto se llama separación de responsabilidades, y es lo que hace al sistema fácil de mantener.

Código — query_engine.py
def submit_message(self, prompt, ...):
    if len(self.mutable_messages) >= \
            self.config.max_turns:
        return TurnResult(
            stop_reason='max_turns_reached')

    projected_usage = self.total_usage\
        .add_turn(prompt, output)
    if projected_usage > max_budget:
        stop_reason = 'max_budget_reached'

    self.compact_messages_if_needed()
En español
📍 Cuando llega un mensaje al motor:
Primero revisa: ¿ya se alcanzó el límite de turnos?
Si sí → para inmediatamente.
 
Luego calcula cuántos tokens costó
esta ida y vuelta.
Si se pasó del presupuesto → frena.
 
Finalmente, si la conversación es muy larga,
recorta los mensajes viejos para liberar espacio.
💡 Concepto: Gestión de contexto

Un context window de un LLM no es infinito. El QueryEngine actúa como un bibliotecario que retira libros viejos del escritorio cuando ya no caben. Sin esta compactación, las conversaciones largas fallarían o se volverían incoherentes.

Módulo 6

El sistema de permisos —
el guardia de seguridad

No todas las herramientas son seguras. El ToolPermissionContext decide quién pasa y quién no.

En el restaurante: El inspector de sanidad con su portapapeles. Tiene dos listas: ingredientes exactos prohibidos ("fugu") y categorías enteras vetadas ("todo lo crudo" si no hay certificación). Si matchea, el platillo no sale.

Imagina un aeropuerto. No importa que tengas boleto — si tu nombre aparece en la lista de no-vuelo, no subes al avión. El sistema de permisos funciona igual: tiene dos listas negras que se revisan antes de ejecutar cualquier herramienta.

🚫
deny_names
Lista de nombres exactos bloqueados. Si la herramienta se llama exactamente "BashTool", no pasa. Como una lista VIP inversa — si estás en ella, estás fuera.
🔒
deny_prefixes
Lista de prefijos bloqueados. Si una herramienta empieza con "mcp_dangerous_", se bloquea automáticamente. Como un filtro de correo que bloquea todo lo que viene de cierto dominio.
Código — permissions.py
@dataclass(frozen=True)
class ToolPermissionContext:
    deny_names: frozenset[str]
    deny_prefixes: tuple[str, ...]

    def blocks(self, tool_name):
        lowered = tool_name.lower()
        return (
            lowered in self.deny_names
            or any(
                lowered.startswith(p)
                for p in self.deny_prefixes
            )
        )
En español
📍 Una ficha de seguridad inmutable con:
→ un conjunto de nombres exactos prohibidos
→ una lista de prefijos prohibidos
 
La función "¿lo bloqueo?":
convierte el nombre a minúsculas
y revisa dos cosas:
① ¿está el nombre exacto en la lista negra?
② ¿empieza con alguno de los prefijos prohibidos?
Si cualquiera es sí → bloqueado.

¿Y dónde se usa esto? En el PortRuntime, justo antes de ejecutar. Mira cómo el runtime revisa herramientas peligrosas:

// runtime.py — Inferencia automática de denegaciones 
def _infer_permission_denials(self, matches):
    denials = []
    for match in matches:
        if match.kind == 'tool' \
           and 'bash' in match.name.lower():
            denials.append(PermissionDenial(
                tool_name=match.name,
                reason='destructive shell execution remains gated'
            ))
    return denials
💡 Patrón: Defense in depth

El sistema tiene dos capas de defensa: el ToolPermissionContext configurable por el usuario (deny_names/deny_prefixes) y la inferencia automática del runtime que bloquea bash sin importar qué. No confías en una sola barrera — como un edificio que tiene tanto cerradura eléctrica como guardia humano.

🧪 Escenario de seguridad
Configuraste deny_prefixes = ["mcp_"] para bloquear todas las integraciones MCP. Un usuario ejecuta un prompt que matchea con "mcp_slack_send". ¿Qué pasa?
Módulo 7

El port a Rust —
el motor nuevo

De prototipo Python a sistema de producción con workspace de crates.

En el restaurante: Empezaron con un food truck (Python — rápido de armar, funcional). Ahora están construyendo una cocina industrial de acero (Rust — imposible de contaminar, con certificaciones de seguridad integradas en la estructura).

El port a Rust no es una traducción línea por línea — es una reimaginación con herramientas que Python no tiene: seguridad de memoria garantizada por el compilador, streaming SSE nativo, y un cliente OAuth completo.

🔌
crates/api
Cliente de la API de Anthropic con soporte de streaming SSE, manejo de errores tipado, y tipos fuertes para requests/responses.
⚙️
crates/runtime
Sesiones, bootstrap, permisos, config, compact, prompt, bash, MCP (stdio + SSE), file_ops, OAuth, remote.
💻
crates/rusty-claude-cli
El binario principal — lo que ejecutas en tu terminal. Un solo ejecutable sin dependencias externas.
📦
crates/commands + tools
Metadata compartida de comandos slash y herramientas built-in, separada de la lógica de ejecución.
// rust/Cargo.toml — Configuración del workspace 
[workspace]
members = ["crates/*"]
resolver = "2"

[workspace.lints.rust]
unsafe_code = "forbid"  ← sin código inseguro, nunca

[workspace.lints.clippy]
all = { level = "warn" }
pedantic = { level = "warn" } ← las reglas más estrictas
🎯 ¿Por qué Rust para un harness?

Un harness de agente maneja ejecución de bash, lectura de archivos, y conexiones de red simultáneamente. En Python, un error de memoria puede corromper estado silenciosamente. En Rust, el compilador te obliga a manejar cada caso posible antes de que el código compile. Es como un inspector de obra que no te deja abrir el edificio hasta que cada conexión eléctrica esté certificada.

🔥 Para llevar

Claw Code demuestra que un harness de agente no es magia — es ingeniería de sistemas con patrones conocidos: catálogos de datos, routers por token, loops de turno con presupuesto, y persistencia de sesión. Cualquiera que entienda estos patrones puede construir (o dirigir) agentes más efectivos.
